================================================================================
OVERALL ASSESSMENT
================================================================================

STRENGTHS
---------
- Scenario-based storytelling (Jordan, Lisa, Carmen, Horizon Tech) is genuinely
  engaging and makes abstract concepts accessible to non-technical readers.
- Deming is woven throughout consistently — not just a name-drop in Chapter 1.
- Chapters 1-4 build well: culture → ownership → psychological safety → process.
- Chapter 4 (Lean Six Sigma) is the strongest chapter. It actually does the math
  (Takt Time, Little's Law) in a narrative context that lands with practitioners.
- The "hidden factory" framing in Chapter 1 is memorable and immediately relatable.
- The book is honest about the failure modes of each methodology — it doesn't
  treat Lean/Agile/ITIL as magic bullets.

WEAKNESSES
----------
1. CHAPTER 5 DOESN'T MEASURE ANYTHING.
   A chapter titled "How Sprints Empower Teams to Deliver Better, Faster" needs
   to define "better" and "faster" with actual formulas. It covers philosophy,
   myths, AI, and tips — but has no measurement section. This is the biggest gap.

2. METRICS ARE CONCENTRATED IN ONE CHAPTER (Ch. 4).
   Little's Law and Takt Time appear only in the ITSM ticketing scenario. They
   are never applied to software delivery, story points, or sprint velocity.
   The concepts translate directly — this is low-hanging fruit.

3. CHAPTER 6 (ITIL) IS A STUB.
   It has the comparison table but no worked example, no scenario resolution,
   and no ITIL-specific metrics (MTTR, MTBF, change success rate). It reads like
   a placeholder that was never filled in.

4. THE "HIDDEN FACTORY" IS NEVER OPERATIONALIZED.
   Introduced powerfully in Chapter 1, rework is never connected to a measurement.
   This is a missed opportunity — the connection is one paragraph away.

5. NO MEASUREMENT CHAPTER.
   A book for practitioners needs at least one chapter or appendix that says:
   "Here is what you measure, here is the formula, here is what good looks like."
   The book describes what to do but stops short of how to know if it's working.

6. CHAPTERS 10 AND 11 OVERLAP SIGNIFICANTLY.
   Both cover AI + methodology integration. Consider merging into one stronger
   chapter with clearer delineation: AI as a tool vs. AI as a risk.

7. AVERAGES EVERYWHERE, NO PERCENTILES.
   The book uses average cycle time, average resolution time throughout.
   In practice, averages are misleading — one outlier skews them.
   The 85th percentile (time within which 85% of work completes) is more
   actionable. One paragraph change, significant practitioner value upgrade.

REAL DASHBOARD vs. BOOK — COMPARISON
================================================================================

DASHBOARD METRICS VALIDATED IN PRODUCTION (as of 2026-03-18)
These are calculated live against ADO data across 5 role-based dashboards:
Dev, BA, QA, Business Partner (BP), Leadership.

Leadership Dashboard:
  Leading Score  = activeDays×2 + commits + prsCreated×3 + prsReviewed×2
                   + comments×1.5  (normalized to 0-100)
  Lagging Score  = cards + storyPoints×2 + taskHours×0.5 (normalized)
  Activity Index = (cards×1 + prsCreated×3 + prsReviewed×2 + commits×1
                   + comments×1.5) × 10 / 10
  PR Turnaround  = Avg(closedDate - creationDate) in hours/days
  Correlation    = |leadingScore - laggingScore|
                   Aligned ≤20 | Watch 20-40 | High Input >40 | High Output <-20

Business Partner Dashboard:
  Impact         = 0.4×StrategicAlignment + 0.4×RiskAvoidance + 0.2×Financial
  Priority       = Impact / Effort
                   Green ≥1.5 | Amber ≥0.8 | Red <0.8
  Effort mapping = XS=1, S=3, M=5, L=7, XL=9
  Completion     = deliveredStories / totalStories × 100
  Stall detection: Story ≥14 days in pre-delivery | Epic ≥30 days | Blocked ≥3

BA Dashboard:
  BA Coverage    = withBA / stories × 100  (target ≥95%)
  Readiness      = readyItems / backlog × 100  (target ≥80%)
  Handoff Lag    = (stateChangedDate - activatedDate) / days  (target <5 days)
  Workload StdDev = sqrt(sum((count - mean)²) / n)
  Health Score   = 0.35×baCov + 0.30×ptRate + 0.20×qaCov + 0.15×(1-staleRate)

QA Dashboard:
  Bug Density    = closedBugs / storiesClosedInPeriod
  Rework Rate    = reworkItems / storiesClosed × 100
  Cycle Time     = createdDate → resolvedDate (work days)
  Buckets        = 0-3d | 4-7d | 8-14d | 15-30d | 31-60d | 60+d

Dev Dashboard:
  APD            = story points closed with valid close date in period
  Cycle Time     = activation → closure (work days), 50th/85th/95th percentile
  Age            = activation → today (work days)
  Stall          = same board column ≥14 days

================================================================================
SPC IMPLEMENTATION GUIDE � LESSONS FROM PR #26392
(For future reference when building SPC charts)
================================================================================

These issues were caught during code review of the leadership dashboard SPC
features (LD-13 cycle time, LD-14 velocity). Document here so we get it right
first time next time.

STATISTICS
----------
1. Always use sample std dev (n-1), not population std dev (n).
   Population formula underestimates sigma on short histories (< 30 points),
   making control limits too tight and flagging normal variation as outliers.
   Code: variance = sumOfSquares / (n - 1)

2. Require n >= 2 before computing stats. Return null and show a clear
   'not enough data' message. Guard all callers against null.

3. Out-of-control means BOTH sides: v > UCL OR (lcl > 0 AND v < LCL).
   LCL is clipped to 0 for cycle time (can't have negative days), so only
   apply the lower bound check when lcl > 0.

CHART.JS MIXED CHARTS (bar + line overlay)
-------------------------------------------
4. Category-scale bar charts (type: 'bar') require line overlay datasets
   to use plain number arrays, NOT {x, y} objects.
   Wrong:  flatLine = val => items.map((_, i) => ({ x: i, y: val }))
   Right:  flatLine = val => items.map(() => val)
   The {x,y} format is for scatter/linear scales only.

5. Scatter charts (type: 'scatter') use {x, y} objects � that is correct
   for the cycle time SPC scatter plot (LD-13).

ADO FIELD NAMES
---------------
6. Activated Date is Microsoft.VSTS.Common.ActivatedDate, NOT System.ActivatedDate.
   System.ActivatedDate does not exist; requesting it returns undefined silently.
   Always verify field names against the ADO REST API docs before coding.

RENDER / CHART LIFECYCLE
-------------------------
7. Every early return in a render function must destroy its chart first.
   Pattern:
     const clearChart = () => {
       if (ldCharts['canvas-id']) { ldCharts['canvas-id'].destroy(); delete ldCharts['canvas-id']; }
     };
     if (!data.length) { clearChart(); showEmptyState(); return; }

8. Show an informative message in the DOM when bailing early � don't just
   return silently. Blank panels with no explanation look like broken code.

ACCESSIBILITY (applies to any new chart canvas)
------------------------------------------------
9. Every <canvas> needs: role=img aria-label=descriptive text
10. Every <th> with tooltip info: use BOTH title= (hover tooltip for mouse
    users) AND aria-label= (screen reader) � replacing one with the other
    removes access for half the audience.
11. New <section> elements need aria-labelledby pointing to their visible
    heading element (add an id to the heading div).
